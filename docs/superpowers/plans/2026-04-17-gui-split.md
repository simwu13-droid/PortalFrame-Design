# GUI File Split Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Split `portal_frame/gui/app.py` (2,566 lines) and `portal_frame/gui/preview.py` (1,640 lines) into focused modules (≤600 lines each) without changing behaviour.

**Architecture:** Module-level functions that take `app` or `canvas` as their first argument. No mixins, no multiple inheritance, no new classes. State continues to live on `PortalFrameApp` and `FramePreview`. Extractions are mechanical: method body moves to a new module, `self.X` becomes `app.X` (or `canvas.X`), and the original method becomes a thin delegate or is removed entirely.

**Tech Stack:** Python 3, tkinter, pytest. No new dependencies.

**Spec:** [`docs/superpowers/specs/2026-04-17-gui-split-design.md`](../specs/2026-04-17-gui-split-design.md)

**Branch:** `refactor/split-large-gui-files` (already created, already checked out).

## Core Extraction Pattern

Every tab/controller/canvas-helper task follows the same pattern. Read this once; each later task references it.

**Before** — a method on `PortalFrameApp`:
```python
class PortalFrameApp(tk.Tk):
    def _build_combos_tab(self, parent):
        self._section_header(parent, "LOAD COMBINATIONS  (AS/NZS 1170.0:2002)")
        combo_text = ("ULS-1: 1.35G ..." )
        tk.Label(parent, text=combo_text, ...).pack(...)
```

**After** — free function in a new module:
```python
# portal_frame/gui/tabs/combos_tab.py
"""Combos tab — read-only display of load combination rules."""

import tkinter as tk
from portal_frame.gui.theme import COLORS, FONT_MONO


def build_combos_tab(app, parent):
    app._section_header(parent, "LOAD COMBINATIONS  (AS/NZS 1170.0:2002)")
    combo_text = ("ULS-1: 1.35G ..." )
    tk.Label(parent, text=combo_text, ...).pack(...)
```

**And the call site in `app.py`** changes from:
```python
self._build_combos_tab(self._tab_pages["Combos"])
```
to:
```python
from portal_frame.gui.tabs.combos_tab import build_combos_tab
...
build_combos_tab(self, self._tab_pages["Combos"])
```

**Mechanical rules for the extraction:**
1. Copy the method body verbatim into the new function.
2. First parameter of the new function is `app` (or `canvas` for `preview.py`).
3. Replace `self.` with `app.` (or `canvas.`) throughout the copied body.
4. Add the `import` for any symbols the moved code uses (`tkinter as tk`, theme constants, widget classes, etc.) at the top of the new file.
5. Delete the original method from `app.py` (or `preview.py`).
6. Replace every call to `self._build_combos_tab(x)` inside `app.py` with `build_combos_tab(self, x)`.

**When an extracted helper needs to call ANOTHER method that is still on the class** (e.g. `self._update_preview()`), that becomes `app._update_preview()` — same rule. Methods stay on the class until their own task extracts them.

**When a tkinter callback is bound to a method** (e.g. `trace_add("write", self._on_frame_change)`), the callback continues to be bound to the class method. The class method body becomes a one-liner delegating to the extracted free function:
```python
def _on_frame_change(self, *args):
    from portal_frame.gui.tabs.frame_tab import on_frame_change
    on_frame_change(self, *args)
```
This preserves the `self._on_frame_change` binding that tkinter holds. Place the `from … import …` inside the method body only if a top-of-file import would cause a cycle; otherwise put it at the top of `app.py`.
